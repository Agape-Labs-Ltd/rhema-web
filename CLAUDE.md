This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`rhema-web` is the public web presence for **Rhema** (Flutter Scripture memorization app by Agape Labs Ltd). It hosts the marketing landing page plus a Bible trivia quiz used for lead capture. Deployed to GitHub Pages at `rhema-app.com` (see `CNAME`).

The project was bootstrapped with Lovable (`https://lovable.dev/projects/ab52e3df-df92-4b55-ac81-4ed38496e345`). Changes pushed to `main` sync back to Lovable, and edits made in Lovable auto-commit here — expect commits from the Lovable bot and don't be surprised by `lovable-tagger` in devDependencies (Vite plugin active only in `development` mode).

## Commands

```bash
npm install          # install deps (npm is used in CI; bun.lockb is also present but CI uses package-lock.json)
npm run dev          # Vite dev server on http://localhost:8080 (host "::", not the default 5173)
npm run build        # production build → dist/
npm run build:dev    # build with development mode (keeps lovable-tagger output)
npm run lint         # eslint . — note: @typescript-eslint/no-unused-vars is disabled
npm run preview      # preview the production build
```

There are no automated tests in this repo — verify changes by running the dev server and exercising the UI.

## Architecture

### Routing & entrypoint
- `src/main.tsx` → `src/App.tsx` mounts `BrowserRouter` with four routes: `/` (`Index` → `RhemaLanding`), `/quiz` (`Quiz`), `/privacy`, `/terms`, and a catch-all `NotFound`. A `ScrollToTop` component resets scroll on route change.
- Path alias: `@/*` resolves to `src/*` (see `vite.config.ts` + `tsconfig.json`). shadcn aliases in `components.json` (`@/components`, `@/components/ui`, `@/lib/utils`, `@/hooks`).
- UI primitives under `src/components/ui/` are the standard shadcn/ui set — prefer extending these over adding new primitives.

### Bible quiz (`/quiz`) — the non-obvious part
Lives in `src/pages/Quiz.tsx` and `src/components/quiz/*`. Three-state machine: `start → playing → score`.

- **Questions** are shipped client-side from `src/questions.json` (~1100 lines). The quiz picks 10 at random and awards `100 + 100 * (timeRemaining / 10)` per correct answer (10-second timer per question).
- **Leaderboard is hourly and partially fake.** `loadLeaderboard()` in `Quiz.tsx` queries Supabase `leaderboard` rows created since the top of the current hour, then merges them with 10 deterministically-generated "seed" players (`SEEDING_NAMES` + a seeded RNG keyed on `startOfHour.getTime()`). The hour boundary resets both the real scores shown and the seed set. When editing leaderboard logic, preserve this seeding — the UI depends on there always being ~10 entries.
- **Lead capture**: on score save, a row goes to `leaderboard` (append-only) and an upsert-by-email happens against `leads` with `high_score` only overwritten if the new score is higher.

### Supabase
- Client: `src/integrations/supabase/client.ts` — auto-generated by Lovable; the header comment says "Do not edit it directly." URL and anon key are hardcoded (public by design). Regenerate via Lovable rather than hand-editing.
- Types: `src/integrations/supabase/types.ts` — likewise generated.
- Schema lives in `supabase/migrations/`. Current schema is two tables, both with permissive RLS (anon can insert to both; anon can select from `leaderboard` and update `leads` by email):
  - `leads(id, created_at, email UNIQUE, name, high_score)`
  - `leaderboard(id, created_at, name, score)` with descending indexes on `created_at` and `score`
- RLS is deliberately open — this is a public, unauthenticated quiz. Don't tighten it without rethinking the flow.

### Landing page
`src/components/RhemaLanding.tsx` uses `framer-motion` (`useScroll` / `useTransform`) for the scroll-driven hero animation (`ContainerScroll` / `AnimatedCard`). The mockup image and logo come from `src/assets/`.

### Styling
Tailwind with a custom theme in `tailwind.config.ts`:
- Fonts: Noto Sans / Noto Serif loaded from Google Fonts in `index.html`.
- Custom color tokens use HSL CSS variables (`--brand-*`, `--teal-*`, `--sacred-text`) defined in `src/index.css` — use the token classes (`bg-brand-primary`, `text-teal-dark`, etc.) rather than raw hex.
- Custom animations: `fade-in`, `scale-in`, `slide-up`, `teal-glow`, `float`, plus gradient utilities `bg-gradient-divine/spiritual/peaceful`.

### Audio
`public/correct.mp3`, `public/incorrect.mp3`, `public/timer_tick.mp3` are referenced from `GameScreen.tsx`. The component explicitly unlocks audio on first user gesture (desktop autoplay policy) — don't collapse the `useRef` initialization without preserving that unlock step.

## Deployment

GitHub Actions workflow `.github/workflows/deploy.yml` runs on push to `main`/`master`:
1. `npm ci` → `npm run build`
2. Copies `CNAME` into `dist/`, then **manually duplicates `dist/index.html` to `dist/404.html`, `dist/privacy.html`, and `dist/terms.html`**. This is the SPA-fallback trick for GitHub Pages and, critically, so the Google Play Store crawler sees 200s at `/privacy` and `/terms` (it doesn't follow client-side routes). If you add new top-level legal/store-required pages, extend this step.
3. Touches `dist/.nojekyll` to disable Jekyll.
4. Publishes to the `gh-pages` branch with `force_orphan: true` and `cname: rhema-app.com`.

If you change the routing structure or add pages that crawlers need to reach directly, update both `App.tsx` **and** the deploy workflow.

## Gotchas

- **Dev server port is 8080**, not Vite's default 5173 (`vite.config.ts`). Scripts/bookmarks that assume 5173 won't work.
- **Two lockfiles exist** (`package-lock.json` and `bun.lockb`). CI uses npm — prefer `npm install` to avoid drift. Only fall back to bun if the user explicitly asks.
- `.env` is gitignored but is not required for local dev — Supabase credentials are baked into `client.ts`.
- The `timer_tick.mp3` file at the repo root is a duplicate of the one in `public/`; the code path uses `/timer_tick.mp3` which is served from `public/`.
- Root catch-all route `*` serves `NotFound`, but GitHub Pages serves `404.html` (a copy of `index.html`) so the SPA can also resolve the route client-side. Both paths must agree.
